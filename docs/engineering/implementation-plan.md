# Implementation Plan

## Goal

把当前 PRD 收敛为一条可落地的开发路线，确保团队先实现最短可验证闭环，而不是同时铺开所有愿景能力。

## Phase 0: Project Setup

目标：

- 确认首版技术栈
- 建立仓库目录骨架
- 固化文档入口和协作规则

完成标准：

- 代码目录结构建立完成
- 根 `AGENTS.md` 与 docs 索引可用
- 团队对 MVP 范围和非目标达成一致

## Phase 1: Project And Material Foundations

目标：

- 支持创建学习项目
- 支持导入资料
- 记录资料处理状态

建议交付：

- 学习项目列表与详情入口
- 文件上传或文本粘贴导入
- 基础资料对象持久化
- 处理状态展示

验收重点：

- 用户能在 1 分钟内创建项目并导入一份资料

## Phase 2: Learning Unit Pipeline

目标：

- 把导入资料转成可浏览学习单元

建议交付：

- 文本抽取流程
- 单元切分逻辑
- 单元标题与顺序生成
- 项目内学习单元列表

验收重点：

- 用户导入后能看到一组基本可读、可顺序浏览的学习单元

## Phase 3: Learning Session And AI Help

目标：

- 打通学习空间中的“阅读 + 讲解 + 问答”

建议交付：

- 学习单元阅读页
- “解释这一段”或“讲解本单元”入口
- 当前单元范围问答
- 原文引用或来源片段展示

验收重点：

- 用户可在当前单元上下文内获得解释，而不是泛化回答

## Phase 4: Quiz And Feedback

目标：

- 为已学单元增加即时验证能力

建议交付：

- 单选、判断、简答三类轻量题型中的最小可用集合
- 提交答案与即时反馈
- 原文依据回显

验收重点：

- 用户能在 3 分钟内完成一次小测并得到反馈

## Phase 5: Review Loop

目标：

- 让学习结果真正回流到后续任务，而不是停在一次性体验

建议交付：

- 基础复习任务生成逻辑
- 错题较多时缩短间隔，表现较好时适当延后
- 按天展示待复习任务
- 掌握状态流转

验收重点：

- 用户在第二天或后续打开系统时能看到待复习内容

## Cross-Cutting Decisions To Make Early

这些决策应在正式编码前确认，否则会影响多个阶段：

1. 首版技术栈与部署形态
2. 文件存储与文本抽取方案
3. 学习单元切分策略
4. AI 提示词与引用格式约束
5. 小测简答题的评分策略
6. 复习任务调度的最小规则集

## Anti-Scope Checklist

如果开发过程中出现以下倾向，应优先回到 PRD 做收缩，而不是继续扩展：

- 想把资料搜索提前做进主链路
- 想自动生成高可信知识图谱
- 想做复杂个性化长期计划
- 想把掌握度设计成精确分数模型
- 想同时适配多个差异很大的学习场景

## Suggested Next Step

如果马上要开始搭工程，推荐按这个顺序推进：

1. 先定技术栈和目录结构。
2. 先做 Phase 1 到 Phase 2，拿到“导入后可学习”的最小体验。
3. 再补 Phase 3 到 Phase 5，完成真正的学习闭环。
