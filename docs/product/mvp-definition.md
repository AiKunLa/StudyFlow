# MVP Definition

## Product Thesis

StudyFlow 的首版产品命题是：

> 把用户已有的静态学习资料，转化为可理解、可提问、可测验、可复习的动态学习流程。

这一定义比“完整学习操作系统”更适合 MVP，因为它能更快验证真实学习价值，也更容易控制实现复杂度。

## Target Users

当前优先聚焦：

- 自学型知识工作者
- 考证型学习者

他们通常具备以下特征：

- 已有 PDF、讲义、Markdown 或笔记类资料
- 对学习目标相对明确
- 需要边学边理解与阶段性验证
- 愿意接受 AI 讲解与问答辅助

## Excluded Users

首版暂不针对：

- K12 低龄用户
- 强课堂协同场景
- 强编程实操训练场景
- 强口语训练场景

## Core MVP Loop

MVP 要确保以下链路完整可用：

1. 创建学习项目
2. 导入一份学习资料
3. 自动切分出学习单元
4. 用户进入一个学习单元并查看原文
5. 用户获得基于原文的 AI 讲解
6. 用户围绕该单元发起问答
7. 用户完成一次轻量小测
8. 系统安排后续复习任务

如果这个闭环不顺畅，其他扩展能力都不应优先开发。

## In Scope

- 学习项目管理
- 文档资料导入
- 文本抽取与学习单元切分
- 基于当前原文的 AI 讲解
- 限定资料范围的知识点问答
- 单元级小测验
- 基于结果的复习任务生成
- 基础掌握状态流转

## Out Of Scope

- 全网资料搜索
- 自动高可信知识依赖图
- 完整长期学习计划
- 精确掌握度评分
- 复杂社交协作
- 多学科深度策略引擎

## Product Rules

以下规则应被视为首版产品约束，而不是实现建议：

- AI 回答必须尽量基于当前学习单元或当前项目资料。
- 学习单元切分追求“基本可读、可用”，不是一次性做成完美知识结构。
- 小测重点验证理解与回忆，不追求复杂考试模拟。
- 掌握度首版用离散状态表达，而不是百分比分数。
- 复习调度首版可简单，但必须真正回流到后续任务视图。

## Success Criteria

### User Success

用户导入一份资料后，能够在一次完整体验中完成：

1. 学习一个学习单元
2. 获取讲解
3. 提问并得到引用原文的回答
4. 完成小测
5. 在后续看到复习任务

### Product Metrics

- 首次资料导入成功率
- 首次学习单元进入率
- AI 讲解使用率
- 单元问答使用率
- 小测完成率
- 次日复习返回率
- 完成“导入 -> 学习 -> 小测 -> 复习”闭环的用户比例

## Key Open Questions

这些问题尚未在仓库中形成最终决策，后续需要单独确认：

1. 首版是否只聚焦某一个更窄的学习场景，例如考证资料。
2. 首版支持的导入格式是否要从 PDF + Markdown 开始，还是更保守。
3. 学习单元的默认粒度按章节、段落，还是混合策略。
4. 小测的简答题是否在首版就做自动评分。
5. 复习任务列表是日历型还是待办型展示。
