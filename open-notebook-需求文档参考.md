## 2. 产品概述

### 2.1 产品名称

Open Notebook

### 2.2 产品定位

一个以隐私优先为核心、支持多模型与多内容类型的 AI 研究助手，目标是成为 NotebookLM 类产品的开源替代方案。

### 2.3 产品愿景

帮助用户将分散的研究材料沉淀为一个可检索、可对话、可总结、可转化、可音频化的个人知识工作台，并让用户对数据、模型和部署方式保持控制权。

### 2.4 核心价值主张

- 数据控制权归用户所有，支持本地部署或私有化部署。
- 不绑定单一 AI 厂商，支持多家模型供应商。
- 支持多模态内容输入与统一知识管理。
- 兼顾探索式对话、结构化提炼、语义搜索和音频消费。
- 提供可扩展 API，便于二次开发和自动化集成。

## 3. 产品要解决的问题

### 3.1 用户痛点

- 研究资料来源分散，难以统一管理。
- 通用大模型无法天然理解用户私有资料。
- 将资料直接上传到第三方 AI 平台存在隐私与合规风险。
- 研究过程中“搜索、阅读、提炼、对比、输出”链路断裂。
- 长文档、音视频、网页等内容的处理成本高，人工整理效率低。
- 缺少把研究成果转成可分享、可复听内容的能力。

### 3.2 产品解决方案

- 以 Notebook 为研究容器组织资料与产出。
- 将 Source、Note、Chat、Search、Transformation、Podcast 串成统一工作流。
- 使用全文上下文和 RAG 两套机制满足不同问答场景。
- 提供细粒度上下文控制，降低隐私暴露与 token 成本。
- 通过多模型配置和凭证系统让用户自主选择 AI 服务。

## 4. 目标用户

### 4.1 核心用户

- 学术研究者
- 行业分析师
- 内容创作者
- 咨询顾问
- 产品经理与技术研究人员

### 4.2 次级用户

- 希望私有部署 AI 知识库的团队
- 需要跨模型对比能力的 AI 重度用户
- 希望基于知识资产生成音频内容的知识型创作者

### 4.3 用户画像

#### 用户画像 A：研究型用户

- 关注资料沉淀、溯源和深度问答
- 高频使用 PDF、网页、论文、会议资料
- 关注引用准确性和结果可验证性

#### 用户画像 B：知识工作者

- 关注快速总结、多资料对比和输出效率
- 需要将 AI 响应沉淀为正式笔记
- 关注信息复用与多项目管理

#### 用户画像 C：私有化部署使用者

- 对数据主权、可控性、厂商切换能力要求高
- 关注本地模型、Ollama、私有 API 网关等接入方式

## 5. 产品目标与边界

### 5.1 产品目标

- 建立完整的研究资料收集、处理、检索、问答、提炼、输出闭环。
- 支持多内容类型进入统一知识体系。
- 支持多模型、多供应商切换。
- 提供可解释、可追溯的 AI 辅助能力。
- 允许用户部署在自己的环境中。

### 5.2 非目标

- 不以社交分享或社区内容分发为核心。
- 不以团队协同编辑作为当前主能力。
- 不以复杂企业权限体系作为现阶段重点。
- 不追求通用云盘或文档协作平台定位。

## 6. 核心概念模型

### 6.1 Notebook

研究项目容器，用于隔离不同主题的资料、笔记、聊天记录和播客产物。

### 6.2 Source

研究输入材料，支持文件、网页链接、纯文本、音视频转写等形式。

### 6.3 Note

研究产出物，可由用户手写，也可由 AI 生成或从回答中保存。

### 6.4 Chat

基于用户选择上下文进行多轮对话，强调探索、追问和上下文连续性。

### 6.5 Ask/Search

面向“一次性问题”与“全局检索”的入口，使用全文检索或向量检索能力，结合 RAG 输出答案。

### 6.6 Transformation

对单个或多个 Source 批量执行的结构化提炼动作，例如摘要、要点抽取、洞察生成。

### 6.7 Podcast

将资料与笔记转化为可播放音频内容，支持多说话人、不同风格和 episode profile。

### 6.8 Context

用户可控制某个 Source 或 Note 对 AI 的暴露级别：

- Full Content
- Summary Only
- Not in Context

## 8. 功能需求

## 8.1 Notebook 管理

### 功能描述

用户可创建、查看、编辑、归档、删除 Notebook，用于区分不同研究主题。

### 核心需求

- 支持创建 Notebook，字段至少包括名称与描述。
- 支持展示 Notebook 列表及其关联的 Source 数量、Note 数量。
- 支持编辑 Notebook 名称、描述、归档状态。
- 支持删除 Notebook，并在删除前给出影响预览：
  - 将删除多少 Note
  - 将解除多少共享 Source 关联
  - 是否删除仅属于该 Notebook 的独占 Source
- 支持按更新时间展示研究项目。

### 验收标准

- 用户可在 1 分钟内创建新 Notebook 并开始导入资料。
- 删除 Notebook 时，系统明确说明级联影响。

---

## 8.2 Source 导入与处理

### 功能描述

用户可将多种内容形式导入系统，并经过抽取、切块、向量化等处理后纳入知识库。

### 支持类型

- 文件上传，如 PDF、Office 文档等
- 网页链接
- YouTube / 视频内容
- 音频内容
- 纯文本粘贴

### 核心需求

- 支持将一个 Source 关联到一个或多个 Notebook。
- 支持 Source 异步处理，返回处理状态和命令 ID。
- 支持处理失败后重试。
- 支持设置是否自动嵌入向量。
- 支持设置处理完成后是否删除原始上传文件。
- 支持展示 Source 状态、标题、标签、是否已嵌入、嵌入分块数。
- 支持下载原始文件。
- 支持查看 Source 详情与全文内容。

### 处理流程要求

1. 抽取原始文本
2. 对长文档切块
3. 为分块生成 embedding
4. 保存元数据、全文、分块和向量
5. 将结果关联到一个或多个 Notebook

### 验收标准

- 常见文件类型可稳定导入。
- 用户能看到处理中的状态反馈，而非长时间无响应。
- 处理失败时可重试而不需要重新创建 Notebook。

---

## 8.3 上下文控制

### 功能描述

用户可以精细控制哪些 Source 和 Note 会被发送给模型，以及发送的粒度。

### 核心需求

- 每个 Source 支持三种上下文级别：全文、摘要、排除。
- 每个 Note 支持上下文配置。
- 聊天区展示总上下文 token 估算。
- 用户切换上下文配置时，后续聊天与问答使用新的上下文。

### 产品价值

- 控制隐私暴露范围
- 控制成本
- 控制答案聚焦程度

---

## 8.4 Note 管理

### 功能描述

用户可记录自己的研究结论，也可保存 AI 输出结果形成二次知识资产。

### 核心需求

- 支持手动创建 Note。
- 支持 Markdown 编辑。
- 支持将 AI 聊天结果保存为 Note。
- 支持将搜索、Transformation 结果保存为 Note。
- 支持 Note 编辑、删除、归属 Notebook。
- 支持区分 Note 类型，如 human / ai。

### 验收标准

- 用户可以把一次高质量 AI 回答沉淀为可长期复用的笔记。

---

## 8.5 Chat 对话

### 功能描述

在 Notebook 语境下，用户围绕选定资料进行多轮对话。

### 核心需求

- 支持创建、切换、管理多个 Chat Session。
- 支持在 Notebook 范围内基于当前上下文进行对话。
- 支持在消息中展示引用信息。
- 支持把 AI 回复保存为 Note。
- 支持展示上下文来源和 token 规模。
- 支持模型选择或使用默认模型。

### 交互原则

- Chat 强调探索式、多轮、上下文连续。
- 用户明确知道 AI 看到了哪些内容。

### 验收标准

- 用户可以对同一批资料持续追问。
- 用户可以快速把有价值答案转成 Note。

---

## 8.6 Search 与 Ask

### 功能描述

用户可跨资料进行全文检索、语义检索和问题驱动式综合问答。

### 核心需求

- 支持 Text Search。
- 支持 Vector Search。
- 支持设置结果数量上限和最低相似度阈值。
- 支持选择是否同时搜索 Sources 与 Notes。
- 支持 Ask 模式：
  - 输入问题
  - 检索相关内容
  - 多阶段生成答案
  - 输出带引用的综合回答
- 支持流式返回答案。

### 搜索模式定义

- Text Search：适用于找关键词、专有名词、具体句子。
- Vector Search：适用于概念查找、近义表达、模糊探索。
- Ask：适用于“一次性拿到综合答案”。

### 验收标准

- 用户能明显区分 Chat 和 Ask 的使用场景。
- 问答结果应附带可核验的来源信息。

---

## 8.7 Transformation 洞察提炼

### 功能描述

用户可基于模板或自定义 Prompt，对 Source 执行结构化提炼。

### 核心需求

- 支持 Transformation 的新增、编辑、删除、列表展示。
- Transformation 字段至少包括名称、标题、描述、Prompt、是否默认启用。
- 支持对单个 Source 发起洞察生成。
- 支持异步生成 Source Insight。
- 支持将 Insight 保存为 Note。
- 支持默认 Prompt 配置。

### 典型场景

- 摘要生成
- 观点提取
- 风险点提取
- 结构化研究笔记
- 多文档统一提炼

### 验收标准

- 用户能复用同一 Transformation 处理多份资料。
- 生成结果能被再次纳入知识体系。

---

## 8.8 Podcast 生成

### 功能描述

用户可将 Notebook 中的 Sources 和 Notes 组织成可播放播客内容。

### 核心需求

- 支持选择要纳入播客的 Sources 与 Notes。
- 支持 Episode Profile：
  - 标题
  - 描述
  - 时长目标
  - 语气 / 风格
  - 关注议题
  - 受众
  - 节目形式
- 支持 Speaker Profile：
  - 名称
  - 专长设定
  - 人格风格
  - 语音模型
  - Voice 配置
- 支持 1 至 4 位说话人。
- 支持播客异步生成和进度跟踪。
- 支持下载音频文件。

### 系统处理流程

1. 分析资料
2. 生成大纲
3. 生成对话脚本
4. 调用 TTS 合成语音
5. 混合导出音频

### 验收标准

- 用户无需阻塞等待，可离开页面后再查看结果。
- 音频生成失败时系统应给出状态和错误说明。

---

## 8.9 模型与凭证管理

### 功能描述

系统支持多供应商接入，并将“供应商凭证”与“可用模型”分离管理。

### 核心需求

- 支持新增、编辑、删除 Credential。
- 支持加密存储 API Key 或接入配置。
- 支持测试连接。
- 支持发现模型。
- 支持注册模型到系统。
- 支持配置默认模型：
  - 默认聊天模型
  - 默认 transformation 模型
  - 大上下文模型
  - 默认 embedding 模型
  - 默认 TTS 模型
  - 默认 STT 模型
  - 默认 tool 模型
- 支持同一供应商下配置多个 Credential。

### 支持的模型类别

- 语言模型
- 向量模型
- 语音转文本模型
- 文本转语音模型

### 支持的供应商类型

- OpenAI
- Anthropic
- Google / Vertex AI
- Groq
- Ollama
- Mistral
- DeepSeek
- xAI
- OpenRouter
- ElevenLabs
- Voyage
- OpenAI Compatible 等

### 验收标准

- 用户无需手工改配置文件，即可在 UI 中完成主流模型接入。

## 8.11 鉴权与安全

### 当前需求

- 系统支持简单密码鉴权中间件。
- API 使用 `Authorization: Bearer <password>` 方式访问。
- 未设置密码时可跳过鉴权。
- 未设置加密密钥时，不允许正常存储凭证。

### 生产建议

- 正式版应替换为 OAuth2 / JWT / API Key 等更完整方案。
- 应支持 HTTPS、CORS 白名单、反向代理限流。

## 9. 关键业务流程

### 9.1 基础研究流程

1. 创建 Notebook
2. 导入一个或多个 Sources
3. 等待内容处理完成
4. 配置上下文级别
5. 发起 Chat 或 Ask
6. 将有价值输出保存为 Note
7. 持续积累、搜索和提炼

### 9.2 批量洞察流程

1. 创建或选择 Transformation
2. 对 Source 执行提炼
3. 查看 Insight
4. 保存为 Note
5. 在后续 Chat / Search 中复用

### 9.3 音频化流程

1. 选择 Notebook 中的 Sources / Notes
2. 选择 Episode Profile 和 Speaker Profile
3. 提交生成任务
4. 轮询任务状态
5. 完成后试听和下载

### 9.4 模型接入流程

1. 新增 Credential
2. 测试连接
3. 发现模型
4. 注册模型
5. 设为默认模型

## 11. 非功能需求

## 11.1 隐私与部署

- 支持自托管部署。
- 默认设计应避免强依赖单一云厂商。
- 用户内容、凭证和模型配置应由用户自主掌控。

## 11.2 性能

- 普通 Notebook/Source/Note 页面操作应保持秒级响应。
- 对话、搜索和问答需支持流式或渐进式反馈。
- 大文件处理与播客生成采用异步任务，避免阻塞主线程。

## 11.3 可扩展性

- 支持新增 AI Provider。
- 支持新增 Source 类型。
- 支持新增 Transformation 模板。
- 支持未来加入团队协作、权限和审计能力。

## 11.4 可维护性

- 前后端分层明确。
- API 模型、服务层、图工作流、数据库迁移应解耦。
- 支持自动迁移数据库结构。

## 11.5 可观测性

- 服务端记录结构化日志。
- 异步任务必须可查询状态。
- 错误需返回统一结构和可理解的信息。

## 11.6 兼容性

- Web 前端应支持现代浏览器。
- 支持 Docker Compose 部署。

---

## 12. 技术实现约束

- 前端：Next.js + React + TypeScript
- 状态管理：Zustand + TanStack Query
- 后端：FastAPI + Pydantic
- AI 编排：LangGraph
- 数据库：SurrealDB
- 向量检索：基于数据库内置 embedding 能力
- 多模型接入：Esperanto 统一封装

这些技术不一定要在你的同类 APP 中原样复用，但在需求文档中至少要明确：

- 是否需要三层架构
- 是否需要异步任务系统
- 是否需要原生向量库能力
- 是否需要多模型抽象层

---

## 13. MVP 范围建议

如果要做一个“类似 Open Notebook 的第一版产品”，推荐 MVP 仅保留以下能力：

- Notebook 管理
- Source 导入
- 文本抽取与向量化
- Chat
- Search / Ask
- Note 保存
- 基础模型配置
- 基础凭证管理

可放到 V2 的能力：

- Podcasts
- 多 Speaker / Episode Profile
- 高级 Transformations
- 多语言完整覆盖
- 大规模运维后台
- 更完善的权限系统

## 15. 同类 APP 编写需求文档时可直接复用的结构

如果你要写一个类似应用的需求文档，建议直接沿用下面结构：

1. 产品定位
2. 用户痛点
3. 用户画像
4. 核心概念模型
5. 信息架构
6. 核心功能模块
7. 关键业务流程
8. 数据对象
9. 非功能需求
10. MVP 与版本规划
11. 成功指标

其中最值得保留的产品思想是：

- 用“研究容器”组织内容，而不是只做一个聊天框
- 用“Source -> Note -> Chat/Ask -> Insight -> Podcast”的链路表达知识流转
- 用“上下文控制”解决隐私、成本、聚焦度三件事
- 用“模型与凭证分离”提升多供应商适配能力

## 16. 对你编写类似 APP 需求文档的建议

如果你后面要写的是“类似产品”而不是“完全复刻产品”，建议你先明确以下取舍：

- 你的产品更偏研究助手、个人知识库，还是内容消费工具
- 是否必须支持多模型与私有化部署
- 是否必须支持播客生成
- 是否需要多 Notebook，还是单工作区即可
- 是否强调引用与可追溯性
- 是否要把搜索、聊天、提炼分成不同能力，而不是都塞进一个聊天入口

如果这些问题先定清楚，你的正式需求文档会比直接照搬功能列表更稳。
